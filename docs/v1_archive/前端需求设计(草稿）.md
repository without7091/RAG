# 二、 前端技术需求设计 (Frontend Design)暂定

前端（管理控制台）不面向最终 C 端提问用户，而是作为**数据资产治理工具**，供管理员进行数据清洗、向量化监控和检索质量测试。

### 1. 架构与视觉规范

- **框架**: Next.js 14+ (App Router), TypeScript
- **UI**: Shadcn/UI, Tailwind CSS, Lucide Icons
- **Editor**: Monaco Editor (配置为 Light Theme)
- **布局风格**：采用现代化管理后台布局（左侧导航树 + 右侧工作区）。以浅色调为主，数据展示注重高信息密度和层级清晰。

### 2. 核心页面规划

#### 2.1 仪表盘主页 (Dashboard)

- **数据大屏**：全局展示当前 Qdrant 数据库的整体状态，包括：总 Collection 数量、各知识集的文档总量、切片 (Chunks) 总数、存储空间占用。
- **任务看板**：实时显示当前正在异步处理的文档转化与入库任务队列。

#### 2.2 知识集管理与数据治理 (Data Governance)

这是系统的核心交互页面。

- **多租户视图**：通过左侧列表切换不同的 `knowledge_base_id`。
- **文件列表区**：
  - 展示该知识集下的所有原始文件，字段包括：`file_name`、`doc_id`、上传时间、处理状态（解析中 / 向量化中 / 已就绪 / 解析失败）。
  - **增量更新入口**：支持对同名或同 `doc_id` 的文件点击“覆盖更新”。系统后台自动执行先删后插的逻辑，前端需提供明确的无冗余覆盖提示。
- **Markdown 审阅器 (可选)**：点击某个已就绪的文档，可在一个只读的 Markdown 编辑器（如 Monaco Editor 实例）中查看经过无损转化后的 `.md` 原始内容及其分块边界，确保版面解析效果符合预期。

#### 2.3 检索调试测试台 (Retrieval Playground)

专为开发者和算法调优设计的测试界面。

- **输入区**：选择指定的知识集，输入测试 Query。
- **过程可视化**：面板实时监听后端 API 推送的 SSE 流，动态展示：`检索触发 -> Qdrant 双路召回 -> Reranker 精排` 的进度条或状态灯。
- **结果透视区 (重点)**：
  - 左侧展示综合拼接后的连续上下文多个片段。
  - 右侧列表平铺渲染底层的 `source_nodes`。每个 Node 必须以卡片形式展示其原始 Markdown 文本、来源文件名称 (Metadata)、以及重排后的置信度得分。支持点击高亮对应文本，实现绝对的知识溯源。
