## 1. 产品定位

**API-First 检索中台**（Retrieval Middleware）—— 面向公司内部 AI 应用的知识检索基础设施。

### 核心原则

- **不包含 LLM 对话/QA 生成**：纯粹的高精度文档检索与上下文合成
- **多租户隔离**：每个知识库（`knowledge_base_id`）对应独立的 Qdrant Collection
- **API-First**：所有功能通过 RESTful API 暴露，前端仅为管理界面
- **Markdown-First**：所有文档先转换为 Markdown 再进行切分和向量化

### 核心能力

```
Upload → Parse → Chunk → Embed → Store → Retrieve → Rerank → Synthesize
```

1. 文档入库：上传 → 解析为 Markdown → 语义切分 → 三路向量生成 → 入库
2. 知识检索：混合检索 → 精排 → 上下文合成 → 返回带溯源的结果

## 4. 数据模型

### 4.1 KnowledgeBase（SQLite ORM）

```python
class KnowledgeBase(Base):
    __tablename__ = "knowledge_bases"

    id: int                        # PK, autoincrement
    knowledge_base_id: str(64)     # UNIQUE INDEX, "kb_{uuid[:16]}"
    knowledge_base_name: str(128)  # 用户设定名称
    description: str(512)          # 描述, nullable
    created_at: datetime           # server_default=now()
    documents: relationship        # One-to-Many → Document (CASCADE)
```

### 4.2 Document（SQLite ORM）

```python
class DocumentStatus(str, Enum):
    UPLOADED  = "uploaded"    # 已上传，未触发向量化
    PENDING   = "pending"     # 等待 Worker 处理
    PARSING   = "parsing"     # 解析中
    CHUNKING  = "chunking"    # 切分中
    EMBEDDING = "embedding"   # 向量化中
    UPSERTING = "upserting"   # 写入向量库
    COMPLETED = "completed"   # 完成
    FAILED    = "failed"      # 失败

class Document(Base):
    __tablename__ = "documents"

    id: int                        # PK, autoincrement
    doc_id: str(128)               # INDEX, SHA256(content)[:32]
    file_name: str(256)            # 原始文件名
    knowledge_base_id: str(64)     # FK → KnowledgeBase.knowledge_base_id (CASCADE)
    status: DocumentStatus         # 处理状态枚举
    chunk_count: int               # 切片数量, default=0
    chunk_size: int | None         # 自定义切片大小 (NULL=用全局默认)
    chunk_overlap: int | None      # 自定义重叠大小
    needs_vector_cleanup: bool     # 重新向量化时需要先清理旧向量
    error_message: str(1024)       # 失败原因
    progress_message: str(256)     # 当前进度描述
    upload_timestamp: datetime     # 上传时间
    updated_at: datetime           # 最后更新时间

    UNIQUE CONSTRAINT: (knowledge_base_id, doc_id)
```

### 4.3 Qdrant Collection 结构

每个知识库对应一个 Qdrant Collection，名称为 `knowledge_base_id`。

```
Collection: "kb_abc123def456"
├── Dense Vector: "dense"
│   ├── Dimension: 1024
│   └── Distance: COSINE
├── Sparse Vector: "sparse"
│   └── Modifier: IDF (服务端 IDF 加权)
├── Sparse Vector: "bm25"
│   └── Modifier: IDF
└── Point Payload:
    ├── text: str              # 切片全文
    ├── doc_id: str            # 文档唯一标识
    ├── file_name: str         # 原始文件名
    ├── knowledge_base_id: str # 知识库 ID
    ├── chunk_index: int       # 切片序号 (0-based)
    ├── header_path: str       # 标题路径 "A > B > C"
    ├── header_level: int      # 标题层级 (0=无标题)
    └── content_type: str      # "text" | "code" | "table"
```

**Request:** `multipart/form-data`

| 参数 | 位置 | 类型 | 必填 | 说明 |
|------|------|------|------|------|
| file | body | File | 是 | 文件 (PDF/DOCX/PPTX/XLSX/MD/TXT) |
| knowledge_base_id | query | string | 是 | 知识库 ID |
| chunk_size | query | int | 否 | 自定义切片大小 |


- `doc_id` = SHA256(file_content)[:32]

### 5.3 Retrieve 端点

#### POST `/api/v1/retrieve` — 核心检索

**Request:**
```json
{
  "user_id": "app_001",
  "knowledge_base_id": "kb_abc123def45678",
  "query": "产品的核心功能有哪些？",
  "top_k": 20,
  "top_n": 3,
  "min_score": 0.1,
  "stream": true
}
```

| 字段 | 类型 | 必填 | 默认 | 说明 |
|------|------|------|------|------|
| user_id | string | 是 | - | 调用方标识 |
| knowledge_base_id | string | 是 | - | 知识库 ID |
| query | string | 是 | - | 查询文本 |
| top_k | int | 否 | 20 | 粗排候选数 (1-100) |
| top_n | int | 否 | 3 | 精排返回数 (1-50) |
| min_score | float | 否 | 0.1 | 最低分数阈值 (0.0-1.0) |
| stream | bool | 否 | true | 是否使用 SSE 流式模式 |

**JSON 模式响应 (stream=false):**
```json
{
  "query": "产品的核心功能有哪些？",
  "knowledge_base_id": "kb_abc123def45678",
  "source_nodes": [
    {
      "text": "[产品概述 > 功能特性]\n\n本产品支持...",
      "score": 0.9234,
      "doc_id": "a1b2c3d4...",
      "file_name": "product-spec.pdf",
      "knowledge_base_id": "kb_abc123def45678",
      "chunk_index": 5,
      "header_path": "产品概述 > 功能特性",
      "context_text": "...(相邻切片合并后的上下文)...",
      "metadata": {
        "header_level": 2,
        "content_type": "text"
      }
    }
  ],
  "total_candidates": 20,
  "top_k_used": 20,
  "top_n_used": 3,
  "min_score_used": 0.1
}
```

**SSE 模式事件流 (stream=true):**

```
event: status
data: {"step": "embedding_query"}

event: status
data: {"step": "hybrid_search"}

event: status
data: {"step": "reranking", "candidates": 20}

event: status
data: {"step": "context_synthesis"}

event: result
data: {"query": "...", "source_nodes": [...], ...}
```

**RetrievalService:**
```python
class RetrievalService:

                       min_score: float | None = None) -> dict
    # 编排: embed → hybrid_search → rerank → filter → synthesize_context
```

**PipelineService:**
```python
class PipelineService:
    async def run_pipeline(self, task_id: str, file_path: str,
                           doc_id: str, file_name: str,
                           knowledge_base_id: str) -> None
    # 阶段: PARSING → CHUNKING → EMBEDDING(dense) → EMBEDDING(sparse) →
    #       BM25 → UPSERTING(delete-before-insert) → COMPLETED
```

**PipelineWorker:**
```python
class PipelineWorker:
    def start(self) -> None
    async def stop(self, timeout: float = 30) -> None
    # 后台轮询 PENDING 文档, Semaphore 并发控制
```

### 6.3 依赖注入

所有服务通过 `app/dependencies.py` 中的工厂函数管理，遵循 Singleton 模式（需要数据库 Session 的服务除外）：

```python
# Singleton 服务（应用生命周期内只创建一次）
get_parsing_service()        -> ParsingService
get_chunking_service()       -> ChunkingService
get_embedding_service()      -> EmbeddingService
get_sparse_embedding_service() -> SparseEmbeddingService
get_reranker_service()       -> RerankerService
get_bm25_service()           -> BM25Service
get_vector_store_service()   -> VectorStoreService  # async (需要 Qdrant client)
get_pipeline_worker()        -> PipelineWorker

# Per-request 服务（每次请求新建，绑定数据库 Session）
get_kb_service_dep()         -> AsyncGenerator[KBService]
get_doc_service_dep()        -> AsyncGenerator[DocumentService]
get_retrieval_service()      -> RetrievalService  # async, 组合多个 Singleton
```

### 9.3 API 客户端类型定义

完整的 TypeScript 类型定义位于 `frontend/src/lib/api.ts`：

```typescript
// 知识库
interface KBInfo { knowledge_base_id, knowledge_base_name, description, document_count, created_at }
interface KBListResponse { knowledge_bases: KBInfo[], total: number }
interface KBCreateRequest { knowledge_base_name: string, description?: string }
interface KBUpdateRequest { knowledge_base_name?: string, description?: string }

// 文档
type DocumentStatus = "uploaded" | "pending" | "parsing" | "chunking" |
                      "embedding" | "upserting" | "completed" | "failed"
interface DocInfo { doc_id, file_name, knowledge_base_id, status, chunk_count,
                    chunk_size, chunk_overlap, error_message, progress_message,
                    upload_timestamp, updated_at }
interface DocListResponse { documents: DocInfo[], total: number }
interface VectorizeDocInfo { doc_id: string, status: DocumentStatus }
interface VectorizeResponse { docs: VectorizeDocInfo[] }
interface ChunkInfo { chunk_index: number, text: string, header_path: string }
interface DocChunksResponse { doc_id, file_name, knowledge_base_id, status,
                              chunk_count, chunks: ChunkInfo[] }

// 检索
interface SourceNode { text, score, doc_id, file_name, knowledge_base_id,
                       chunk_index, header_path, context_text, metadata }
interface RetrieveRequest { user_id, knowledge_base_id, query, top_k?, top_n?, stream? }
interface RetrieveResponse { query, knowledge_base_id, source_nodes: SourceNode[],
                             total_candidates, top_k_used, top_n_used, min_score_used }

// 统计
interface StatsResponse { total_knowledge_bases, total_documents, total_chunks,
                          knowledge_bases: Array<{ knowledge_base_id, knowledge_base_name,
                          document_count }> }
```

### 9.4 关键交互逻辑

**文档处理状态轮询：** 当存在 processing 状态的文档时，每 3 秒自动刷新文档列表。

**SSE 检索流程：** 使用 `ReadableStream` 解析 SSE 事件流，4 步骤实时可视化：
1. Query 向量化 (`embedding_query`)
2. 混合检索 (`hybrid_search`)
3. 重排序 (`reranking`)
4. 构建响应 (`building_response` / `context_synthesis`)

**进度条映射：**

| 状态 | 进度 |
|------|------|
| uploaded | 0% |
| parsing | 15% |
| chunking | 35% |
| embedding | 60% |
| upserting | 85% |
| completed | 100% |
