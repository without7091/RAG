### 关键约束

- **单 Worker 进程**：Qdrant 本地存储 + SQLite 不支持多进程并发访问
- **全异步**：所有 I/O 操作使用 `async/await`，绝不阻塞事件循环
- **并发控制层级**：Pipeline 级（Semaphore=2）→ Embedding 级（Semaphore=5）

## 2. PipelineWorker 后台处理

### 设计模式：Poll-Dispatch-Execute

文件: `backend/app/services/pipeline_worker.py`

```python
class PipelineWorker:
    def __init__(self, session_factory, max_concurrency=2, poll_interval=2.0):
        self._semaphore = asyncio.Semaphore(max_concurrency)
        self._poll_interval = poll_interval
        self._active_tasks: set[asyncio.Task] = set()
        self._active_doc_keys: set[tuple[str, str]] = set()  # (kb_id, doc_id)
```

#### Poll（轮询）

```python
async def _poll_loop(self) -> None:
    while self._running:
        try:
            await self._poll_and_dispatch()
        except Exception:
            logger.exception("Error in pipeline worker poll cycle")
        await asyncio.sleep(self._poll_interval)
```

- 每 `poll_interval`（默认 2 秒）轮询一次数据库
- 异常不中断轮询循环，记录日志后继续

#### Dispatch（分发）

```python
async def _poll_and_dispatch(self) -> None:
    available = self._max_concurrency - len(self._active_doc_keys)
    if available <= 0:
        return

    async with self._session_factory() as session:
        result = await session.execute(
            select(Document)
            .where(Document.status == DocumentStatus.PENDING)
            .limit(available)
        )
        pending_docs = list(result.scalars().all())

    for doc in pending_docs:
        doc_key = (doc.knowledge_base_id, doc.doc_id)
        if doc_key in self._active_doc_keys:
            continue  # 防止重复处理

        self._active_doc_keys.add(doc_key)
        task = asyncio.create_task(self._run_pipeline_wrapper(...))
        self._active_tasks.add(task)
        task.add_done_callback(self._active_tasks.discard)
        task.add_done_callback(lambda t, k=doc_key: self._active_doc_keys.discard(k))
```

**关键设计：**

1. **可用槽位计算**：`max_concurrency - active` 决定本轮最多拾取几个文档
2. **去重保护**：`_active_doc_keys` 集合确保同一文档不会被重复拾取
3. **回调清理**：任务完成后自动从 `_active_tasks` 和 `_active_doc_keys` 中移除

#### Execute（执行）

```python
async def _run_pipeline_wrapper(self, kb_id, doc_id, file_name, ...):
    async with self._semaphore:  # 并发控制
        async with self._session_factory() as session:
            doc_svc = DocumentService(session)
            doc = await doc_svc.get_by_doc_id_and_kb(doc_id, kb_id)

            # 二次校验状态
            if doc is None or doc.status != DocumentStatus.PENDING:
                return

            # 处理向量清理（重新向量化场景）
            if doc.needs_vector_cleanup:
                vs = await get_vector_store_service()
                await vs.delete_by_doc_id(doc.knowledge_base_id, doc.doc_id)
                doc.needs_vector_cleanup = False
                await session.commit()

            # 执行 Pipeline
            pipeline = PipelineService(session, ...)
            await pipeline.run_pipeline(...)
```

**关键设计：**

1. **Semaphore 双重保护**：虽然 Dispatch 已控制拾取数量，Semaphore 提供额外的并发上限保证
2. **二次状态校验**：获取 Semaphore 后重新查询文档状态，防止轮询间隙中状态被外部改变
3. **向量清理标记**：`needs_vector_cleanup=True` 时先删除旧向量再执行新的 Pipeline
4. **独立 Session**：每个 Pipeline 任务使用独立的数据库 Session

### 优雅停机

```python
async def stop(self, timeout: float = 30) -> None:
    self._running = False

    # 1. 取消轮询循环
    if self._poll_task is not None:
        self._poll_task.cancel()
        await self._poll_task  # 等待 CancelledError

    # 2. 等待活跃任务完成（最多 timeout 秒）
    if self._active_tasks:
        done, pending = await asyncio.wait(self._active_tasks, timeout=timeout)
        # 超时后强制取消
        for task in pending:
            task.cancel()
        if pending:
            await asyncio.wait(pending, timeout=5)
```

停机流程：停止轮询 → 等待进行中的任务 → 超时强制取消 → 清理。

## 3. Embedding 批量处理与自动降级

### 正常流程：批量发送

文件: `backend/app/services/embedding_service.py`

```python
async def embed_texts(self, texts: list[str]) -> list[list[float]]:
    for i in range(0, total, self.batch_size):  # batch_size=64
        batch = texts[i : i + self.batch_size]
        result = await self._embed_batch(batch)
        all_embeddings.extend(result)
```

单次 API 调用最多发送 64 个文本，减少 HTTP 往返。

### HTTP 400 自动降级

部分 Embedding API 对特定输入（超长文本、特殊字符）返回 HTTP 400。降级策略：

```python
async def _embed_batch(self, batch: list[str]) -> list[list[float]]:
    if self._fallback_to_single:
        return await self._embed_one_by_one(batch)

    try:
        return await self._call_api(batch)
    except EmbeddingError as e:
        if "400" in str(e) and len(batch) > 1:
            logger.warning(
                "Embedding API rejected batch of %d texts (400). "
                "Falling back to one-by-one mode.",
                len(batch),
            )
            self._fallback_to_single = True  # 全局标记，后续请求也走逐条模式
            return await self._embed_one_by_one(batch)
        raise
```

### 逐条模式并发控制

降级后不是串行发送，而是通过 Semaphore 控制并发：

```python
async def _embed_single(self, text: str) -> list[float]:
    async with self._concurrency_sem:  # Semaphore(5)
        results = await self._call_api([text])
        return results[0]

async def _embed_one_by_one(self, texts: list[str]) -> list[list[float]]:
    tasks = [self._embed_single(t) for t in texts]
    return list(await asyncio.gather(*tasks))
```

- `asyncio.gather` 并发发送所有请求
- `Semaphore(embedding_concurrency=5)` 限制同时在途的请求数
- 相比串行模式，吞吐量提升约 5x

### Sparse Embedding 同样的降级逻辑

`SparseEmbeddingService` 复用了相同的模式（`backend/app/services/sparse_embedding_service.py:103-119`），确保 API 模式下的稀疏向量生成也具有同样的容错能力。

### 前端消费

文件: `frontend/src/lib/api.ts:264-324`

```typescript
export function retrieveSSE(data: RetrieveRequest, handlers: {
    onStatus?: (step: string, candidates?: number) => void;
    onResult?: (result: RetrieveResponse) => void;
    onError?: (error: string) => void;
}) {
    const ctrl = new AbortController();

    (async () => {
        const res = await fetch(`${BASE}/retrieve`, {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify({ ...data, stream: true }),
            signal: ctrl.signal,
        });

        const reader = res.body.getReader();
        const decoder = new TextDecoder();
        let buffer = "";

        while (true) {
            const { done, value } = await reader.read();
            if (done) break;
            buffer += decoder.decode(value, { stream: true });

            const lines = buffer.split("\n");
            buffer = lines.pop() || "";

            let currentEvent = "";
            for (const line of lines) {
                if (line.startsWith("event:")) {
                    currentEvent = line.slice(6).trim();
                } else if (line.startsWith("data:")) {
                    const json = JSON.parse(line.slice(5).trim());
                    if (currentEvent === "status") handlers.onStatus?.(json.step, json.candidates);
                    else if (currentEvent === "result") handlers.onResult?.(json);
                    else if (currentEvent === "error") handlers.onError?.(json.error);
                }
            }
        }
    })();

    return () => ctrl.abort();  // 返回取消函数
}
```

**关键设计：**

1. **ReadableStream**：使用 `res.body.getReader()` 逐块读取，无需等待完整响应
2. **Buffer 机制**：处理 chunk 边界可能切断 SSE 行的情况
3. **AbortController**：返回取消函数，用户离开页面时可终止请求
4. **事件分发**：通过 `event:` 行确定事件类型，`data:` 行解析 JSON

## 5. 启动恢复机制

### 问题

如果应用在文档处理过程中（PARSING/CHUNKING/EMBEDDING/UPSERTING）崩溃或重启，这些文档会永远卡在中间状态。

### 解决方案

在应用启动时（`lifespan` 上下文），将所有卡在处理状态的文档重置为 PENDING：

```python
# backend/app/main.py:43-54
@asynccontextmanager
async def lifespan(app: FastAPI):
    # ...初始化...

    # 启动恢复: 重置卡住的文档
    engine = get_engine()
    async with engine.begin() as conn:
        result = await conn.execute(
            text(
                "UPDATE documents SET status = 'PENDING', needs_vector_cleanup = 1 "
                "WHERE status IN ('PARSING', 'CHUNKING', 'EMBEDDING', 'UPSERTING')"
            )
        )
        if result.rowcount > 0:
            logger.info("Startup recovery: reset %d stuck document(s) to PENDING", result.rowcount)

    # 启动 Worker
    worker = get_pipeline_worker()
    worker.start()
```

**关键细节：**

- `needs_vector_cleanup = 1`：标记需要清理旧向量，因为 EMBEDDING 或 UPSERTING 阶段可能已经写入了部分向量数据
- Worker 在后续拾取这些文档时，会先执行 `delete_by_doc_id()` 清理残留向量，再重新执行完整 Pipeline
- 使用原生 SQL 而非 ORM，因为 SQLAlchemy Enum 存储的是大写名称

## 6. 重试机制

### 重试装饰器

文件: `backend/app/utils/retry.py`

基于 `tenacity` 库，提供两级重试策略：

#### 通用 API 重试

```python
def retry_on_api_error(max_attempts=3, min_wait=1, max_wait=10):
    return retry(
        stop=stop_after_attempt(max_attempts),
        wait=wait_exponential(multiplier=1, min=min_wait, max=max_wait),
        retry=(
            retry_if_exception_type((TimeoutError, ConnectionError, OSError))
            | retry_if_rate_limit()
        ),
    )
```

- **触发条件**：网络错误（超时、连接失败）+ HTTP 429（限流）
- **退避策略**：指数退避 `min(1 * 2^n, 10)` 秒
- **最大重试**：3 次
- **应用位置**：`EmbeddingService._call_api()`, `SparseEmbeddingService._call_api()`, `RerankerService.rerank()`

#### 限流专用重试

```python
def retry_on_rate_limit(max_attempts=5, min_wait=2, max_wait=30):
    return retry(
        stop=stop_after_attempt(max_attempts),
        wait=wait_exponential(multiplier=2, min=min_wait, max=max_wait),
        retry=retry_if_rate_limit(),
    )
```

- **触发条件**：仅 HTTP 429
- **退避策略**：更长的退避 `min(2 * 2^n, 30)` 秒
- **最大重试**：5 次
- **用途**：高频场景下的限流保护

### 限流检测逻辑

```python
class retry_if_rate_limit(retry_base):
    def __call__(self, retry_state):
        exc = retry_state.outcome.exception()
        return isinstance(exc, (EmbeddingError, RerankerError)) and "429" in str(exc)
```

通过检查异常消息中是否包含 "429" 判断是否为限流错误，兼容不同 API 返回格式。

### 重试时序示例

```
请求 1: EmbeddingService._call_api() → HTTP 429
等待 1s
请求 2: EmbeddingService._call_api() → HTTP 429
等待 2s
请求 3: EmbeddingService._call_api() → 成功

请求 X: EmbeddingService._call_api() → TimeoutError
等待 1s
请求 X+1: EmbeddingService._call_api() → TimeoutError
等待 2s
请求 X+2: EmbeddingService._call_api() → TimeoutError
抛出异常 → Pipeline 标记为 FAILED
```

## 7. Delete-before-Insert 一致性保证

### 场景分析

当用户对同一文档重新向量化时，需要确保：

1. 旧向量被完全清除
2. 新向量完整写入
3. 不会出现新旧混合的中间状态

### 实现机制

#### 触发路径 1：首次向量化

```
Vectorize API → status=PENDING → Worker拾取 → Pipeline:
  delete_by_doc_id()  ← 即使是首次，也执行删除（幂等操作）
  upsert_points()     ← 写入新向量
```

#### 触发路径 2：重新向量化（Retry）

```
Retry API → status=PENDING, needs_vector_cleanup=True → Worker拾取:
  # _run_pipeline_wrapper 中
  if doc.needs_vector_cleanup:
      vs.delete_by_doc_id(kb_id, doc_id)  ← 预清理
      doc.needs_vector_cleanup = False
      session.commit()

  # Pipeline 中再次执行
  delete_by_doc_id()  ← 第二次删除（双重保证）
  upsert_points()
```

#### 触发路径 3：启动恢复

```
启动时检测卡住的文档 → status=PENDING, needs_vector_cleanup=True
→ 与路径 2 相同
```

### 一致性窗口

在 `delete_by_doc_id()` 和 `upsert_points()` 之间有一个短暂的窗口，文档的向量数据为空。这个窗口通常只有几十毫秒（Qdrant 本地操作延迟极低）。在此期间如果有检索请求命中该文档，结果会暂时缺失该文档的切片，但不会返回错误数据。

## 8. 全局 HTTP 客户端生命周期管理

### httpx.AsyncClient

# backend/app/core/httpx_client.py
_httpx_client: httpx.AsyncClient | None = None

async def get_httpx_client() -> httpx.AsyncClient:
    global _httpx_client
    if _httpx_client is None:
        _httpx_client = httpx.AsyncClient(timeout=60.0)
    return _httpx_client

async def close_httpx_client():
    global _httpx_client
    if _httpx_client is not None:
        await _httpx_client.aclose()
        _httpx_client = None

- 全局单例，所有服务共享同一个连接池

### 生命周期管理

```python
# backend/app/main.py
@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动：初始化 DB → 恢复 stuck docs → 启动 Worker
    await init_db()
    worker.start()

    yield  # 应用运行中

    # 关闭：停止 Worker → 关闭 HTTP 客户端 → 关闭 Qdrant → 关闭 DB
    await worker.stop(timeout=30)
    await close_httpx_client()
    await close_qdrant_client()
    await close_db()
```

关闭顺序：
1. **Worker 先停**：确保没有新的 API 调用
2. **HTTP 客户端**：所有外部 API 调用入口
3. **Qdrant 客户端**：向量存储
4. **数据库**：最后关闭元数据库

这个顺序确保不会在依赖的资源已关闭后，仍有活跃的操作尝试使用它。
