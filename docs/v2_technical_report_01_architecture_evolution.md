# v2 技术报告（一）：架构演进与 v1 差异总览

## 1. 结论摘要

当前仓库的 `v2` 不是对 `v1` 的推倒重来，而是在 **保留 v1 核心 RAG 主链路** 的基础上，新增了三类关键能力：

1. **知识治理能力增强**：知识库从平铺列表升级为固定两层目录树；
2. **检索能力增强**：检索从“混合召回 + 精排”升级为“查询改写 fanout + 合并去重 + 调试信息 + SSE 状态流”；
3. **运行稳定性增强**：模型调用超时、并发、重试、启动恢复、Worker 补偿重试全部配置化。

从代码事实看，v2 的定位更准确地说是：

- **v1 的生产化增强版**；
- 而不是一个全新的检索引擎版本。

---

## 2. v1 基线能力回顾

结合 `docs/v1_archive/` 中的存档文档，v1 已经具备以下骨架能力：

- Markdown-First 解析策略；
- 结构化切分与 chunk metadata；
- Dense + Sparse + BM25 三路混合检索；
- Qdrant 混合向量存储；
- Reranker 精排；
- 上下文窗口合成；
- 异步 Worker 驱动的文档入库；
- 基于文档状态的前端轮询。

因此，v2 的价值不在“第一次实现 RAG”，而在“把已有 RAG 主链路做成更像平台”的形态。

---

## 3. v2 相比 v1 的核心变化

| 维度 | v1 特征 | v2 当前实现 | 价值 |
|---|---|---|---|
| 知识组织 | 以知识库平铺为主 | 新增 `kb_folders` 两层目录树、树形查询、知识库迁移目录 | 适合项目/业务域分层治理 |
| 检索入口 | 以单 query 检索为主 | 新增 query rewrite、fanout 检索、候选合并、debug 字段 | 召回覆盖面和可解释性更强 |
| 响应方式 | 以普通 JSON 为主 | 检索支持 JSON / SSE 双模式 | 便于前端显示检索阶段进度 |
| 稳定性 | 共享超时/基础异常处理 | 模型调用超时拆分、并发闸门、可恢复异常重试、Worker 补偿重试 | 更适合内网长耗时模型 |
| 可运维性 | 依赖基础日志和状态流转 | 启动恢复、`needs_vector_cleanup`、更细的错误分级 | 更利于失败恢复和重复向量化 |
| 数据治理 | 文件上传与向量化 | 新增预切分上传、文档参数更新、下载、重试、chunk 预览 | 更像面向运营的治理后台 |
| 前端交互 | 基础管理页 | 目录树、选中态保持、SSE 进度反馈、治理页轮询增强 | 更贴近运营使用场景 |

---

## 4. 当前 v2 的后端架构视图

### 4.1 分层结构

- **API 层**：`backend/app/api/v1/`
  - `kb.py`：知识库与目录管理
  - `document.py`：上传、预切分上传、向量化、重试、下载、设置
  - `retrieve.py`：检索 JSON / SSE 双模式
  - `stats.py`：全局统计
- **Schema 层**：`backend/app/schemas/`
  - 对外请求/响应结构已经比较完整，前端 `frontend/src/lib/api.ts` 与其保持同步
- **Service 层**：`backend/app/services/`
  - `pipeline_service.py` / `prechunk_pipeline_service.py`
  - `retrieval_service.py`
  - `query_rewrite_service.py`
  - `embedding_service.py` / `reranker_service.py` / `chat_completion_service.py`
  - `vector_store_service.py`
- **持久化层**
  - SQLite：知识库、目录、文档状态
  - Qdrant：chunk payload、dense/sparse/bm25 向量

### 4.2 生命周期编排

当前 v2 把系统分成两条主链：

1. **入库链**
   - 创建知识库/目录
   - 上传原始文件或预切分 JSON
   - 显式触发向量化
   - Worker 后台消费 `PENDING` 文档
2. **查询链**
   - 可选 query rewrite
   - 多 query fanout 检索
   - 候选合并
   - 可选 rerank
   - 可选上下文合成
   - JSON 或 SSE 返回

这比 v1 更平台化：上传、处理、查询、治理已经明显解耦。

---

## 5. v2 已经落地的关键增强

### 5.1 知识库目录树

`v2` 新增了 `kb_folders` 模型和两层目录约束：

- 根目录深度为 `1`
- 知识库只能挂到深度为 `2` 的叶子目录
- `GET /api/v1/kb/tree` 直接返回树结构

这让“知识资产治理”从简单列表升级为“目录 + 知识库 + 文档”的三层视图。

### 5.2 检索链路可观测性

`RetrieveRequest` 和 `RetrieveResponse` 在 v2 中明显扩展，新增了：

- `enable_query_rewrite`
- `query_rewrite_debug`
- `enable_context_synthesis`
- `debug.query_plan`
- `debug.candidate_stats`
- SSE `status/result/error` 事件

这意味着 v2 的检索接口不再只是“返回命中文本”，而是逐步具备“解释检索过程”的能力。

### 5.3 内网模型稳定性治理

`v2.1.0` 已把稳定性治理做进代码，而不只是写在方案里：

- 通用 HTTP 超时拆分为 connect/read/write/pool；
- Query Rewrite 使用独立超时与并发；
- Embedding / Reranker / Query Rewrite 都引入并发闸门；
- 可恢复异常支持自动重试；
- `PipelineWorker` 对可恢复上游异常做文档级补偿重试；
- 启动时会把卡在处理中间态的文档恢复为 `PENDING`。

这说明 v2 已经不只是“功能可用”，而是开始处理真实部署环境中的失败恢复。

---

## 6. v2 保持不变或延续 v1 的部分

以下能力在 v2 中仍然延续 v1 思路，没有发生根本性范式变化：

- 解析策略仍然是 Markdown-First；
- chunk 仍以结构化切分和 metadata 为核心；
- Qdrant 仍是单集合对应单知识库；
- 检索主干仍然是 Dense + Sparse + BM25 + RRF + Rerank；
- 文档处理仍然是“上传/触发/后台消费”的异步模型；
- 前端仍然通过文档状态轮询掌握处理进度。

因此，v2 的真实技术路线是“延续既有路线并补平台能力”，不是“替换底层检索范式”。

---

## 7. 对 v1 到 v2 的判断

### 7.1 从“RAG 功能”走向“RAG 平台”

v1 更像：

- 已可工作的 RAG 检索中台；
- 重点在解析、切分、向量化、混合检索。

v2 更像：

- 可被运营、可被调试、可被其他系统稳定调用的知识服务平台；
- 重点在治理、可观测性、稳定性、接口契约。

### 7.2 从“单路径处理”走向“分层治理”

v2 新增的目录树、预切分上传、文档设置更新、重试下载等能力，本质上是在做：

- 数据治理平面；
- 检索执行平面；
- 运维恢复平面。

这三个平面是 v1 文档里有规划，但在 v2 代码中才更完整落地的部分。

---

## 8. 当前版本的总体评价

从代码结构、测试分布、前后端契约和版本说明综合判断，当前 v2 有三个明显特征：

- **主链路完整**：上传、入库、检索、治理、前端展示已打通；
- **增强点明确**：目录树、查询改写、SSE、重试恢复都有明确落点；
- **平台边界仍待补强**：认证授权、OpenAPI 契约完整性、分布式任务控制、文件存储策略仍有 v3 空间。

一句话概括：

> v2 已经从“可运行的 RAG Demo/中台”进化为“具备治理和稳定性能力的 RAG 平台雏形”，但离真正面向多系统接入的企业级平台还差一层接口治理与安全治理。

---

## 9. 面向 v3 的总方向

如果 v3 继续沿当前路线演进，建议重点放在四个主题：

1. **接口治理**：补齐 OpenAPI、错误码、流式契约；
2. **安全治理**：认证、授权、租户隔离、审计；
3. **一致性治理**：文件存储命名、重入向量化、批量事务边界；
4. **规模治理**：分布式 Worker、持久化重试、分页与批量性能优化。

这些内容将在后续报告和风险清单中展开。
