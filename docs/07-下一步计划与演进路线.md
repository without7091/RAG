## 2. 短期优化（1-2 个迭代）

### 2.1 Embedding 缓存

**问题：** 同一文档重新向量化时，所有切片重新调用 Embedding API，浪费算力和时间。

**方案：** 对切片文本做 hash，缓存 `hash → embedding vector` 映射。

```python
# 伪代码
class EmbeddingCache:
    def __init__(self, storage_path: str):
        self._cache = SqliteDict(storage_path)  # 或 Redis

    async def get_or_embed(self, texts: list[str], embed_fn) -> list[list[float]]:
        results = [None] * len(texts)
        miss_indices = []

        for i, text in enumerate(texts):
            key = hashlib.md5(text.encode()).hexdigest()
            cached = self._cache.get(key)
            if cached is not None:
                results[i] = cached
            else:
                miss_indices.append(i)

        if miss_indices:
            miss_texts = [texts[i] for i in miss_indices]
            new_embeddings = await embed_fn(miss_texts)
            for i, emb in zip(miss_indices, new_embeddings):
                results[i] = emb
                key = hashlib.md5(texts[i].encode()).hexdigest()
                self._cache[key] = emb

        return results
```

**收益：** 重新向量化时，未修改的切片直接命中缓存，只有新增/修改的切片需要调用 API。

### 2.2 Qdrant Payload 索引

**问题：** `delete_by_doc_id` 和 `get_chunks_by_doc_id` 使用 Payload 过滤，未建索引时需要全量扫描。

**方案：** 在创建 Collection 时为高频过滤字段建立 Payload 索引。

```python
async def create_collection(self, collection_name: str) -> None:
    await self.client.create_collection(...)

    # 为 doc_id 建立 keyword 索引
    await self.client.create_payload_index(
        collection_name=collection_name,
        field_name="doc_id",
        field_schema=models.PayloadSchemaType.KEYWORD,
    )
    # 为 chunk_index 建立 integer 索引
    await self.client.create_payload_index(
        collection_name=collection_name,
        field_name="chunk_index",
        field_schema=models.PayloadSchemaType.INTEGER,
    )
```

**收益：** `doc_id` 过滤从 O(N) 降低到 O(1) 级别，显著加速删除和查询操作。

### 2.3 API Key 认证

**问题：** 当前 API 完全开放，无鉴权。

**方案：** 通过 FastAPI 依赖注入实现 Header-based API Key 认证。

```python
# app/auth.py
from fastapi import Security, HTTPException
from fastapi.security import APIKeyHeader

api_key_header = APIKeyHeader(name="X-API-Key")

async def verify_api_key(key: str = Security(api_key_header)):
    valid_keys = get_settings().api_keys  # 从配置读取
    if key not in valid_keys:
        raise HTTPException(status_code=401, detail="Invalid API key")
    return key
```

**配置：**
```env
API_KEYS=key1,key2,key3
```

### 2.4 健康检查端点

**方案：** 添加 `/health` 和 `/ready` 端点。

```python
@app.get("/health")
async def health():
    return {"status": "ok"}

@app.get("/ready")
async def ready():
    # 检查核心依赖
    checks = {}
    checks["database"] = await check_db()
    checks["qdrant"] = await check_qdrant()
    checks["worker"] = worker.is_running()
    all_ok = all(checks.values())
    return {"status": "ready" if all_ok else "degraded", "checks": checks}
```

## 3. 中期增强（3-5 个迭代）

### 3.1 分布式 Qdrant

**问题：** 本地磁盘模式不支持水平扩展和高可用。

**方案：** 部署独立的 Qdrant Server，支持分片和副本。

```yaml
# docker-compose.yml 增加 Qdrant 服务
services:
  qdrant:
    image: qdrant/qdrant:v1.12.0
    ports:
      - "6333:6333"
      - "6334:6334"
    volumes:
      - qdrant-data:/qdrant/storage
    environment:
      - QDRANT__SERVICE__GRPC_PORT=6334
```

**配置变更：**
```env
# 从本地路径切换为远程地址
QDRANT_URL=http://qdrant:6333
# 移除 QDRANT_STORAGE_PATH
```

**代码变更：** `get_qdrant_client()` 从 `path=` 模式切换到 `url=` 模式。

### 3.2 RBAC 权限控制

**方案：** 基于角色的访问控制。

| 角色 | 权限 |
|------|------|
| admin | 所有操作 |
| editor | 创建/编辑 KB，上传/删除文档，向量化 |
| viewer | 列表查看，检索 |
| api_user | 仅检索接口 |

**实现思路：**
- JWT Token 携带角色信息
- FastAPI 依赖注入检查权限
- 前端根据角色显示/隐藏操作按钮

### 3.3 Prometheus 监控

**方案：** 暴露 `/metrics` 端点，接入 Prometheus + Grafana。

```python
from prometheus_client import Counter, Histogram, make_asgi_app

# 核心指标
retrieve_requests = Counter("rag_retrieve_total", "Total retrieve requests", ["kb_id"])
retrieve_latency = Histogram("rag_retrieve_seconds", "Retrieve latency")
pipeline_duration = Histogram("rag_pipeline_seconds", "Pipeline processing duration", ["stage"])
embedding_api_errors = Counter("rag_embedding_errors_total", "Embedding API errors", ["error_type"])
active_pipelines = Gauge("rag_active_pipelines", "Currently processing documents")
```

**关键监控面板：**
- 检索 QPS 和延迟分布
- Pipeline 各阶段耗时
- Embedding API 错误率和重试率
- 活跃 Pipeline 数量
- 知识库/文档/切片总量

### 3.4 查询日志与分析

**方案：** 记录每次检索请求及结果，用于质量分析和调优。


### 3.5 数据库迁移到 PostgreSQL

**问题：** SQLite 不支持并发写入和多副本部署。

**方案：** 使用 PostgreSQL + asyncpg。

```env
# 从 SQLite
SQLITE_DATABASE_URL=sqlite+aiosqlite:///./data/metadata.db
# 迁移到 PostgreSQL
DATABASE_URL=postgresql+asyncpg://user:pass@postgres:5432/rag
```

**收益：**
- 支持多 Worker 进程并发写入
- 支持数据库级别的事务和锁
- 为后续分布式部署铺路

## 4. 长期演进（6+ 个迭代）

### 4.1 多模态支持

**扩展支持的文档类型：**

| 格式 | 技术方案 | 说明 |
|------|---------|------|
| 扫描版 PDF | Tesseract / PaddleOCR | OCR 识别后转为文本 |
| 图片 (.jpg/.png) | 多模态 Embedding 或 OCR | 图文混合检索 |
| 音频 (.mp3/.wav) | Whisper ASR | 语音转文字后入库 |
| 视频 (.mp4) | Whisper + 关键帧提取 | 字幕和画面内容 |

**架构变更：** `ParsingService` 扩展为策略模式，根据文件类型选择不同的解析器。

### 4.2 高级切分策略

#### 语义切分（Semantic Chunking）

基于 Embedding 相似度检测段落边界：

```
段落1 ←相似→ 段落2 ←不相似→ 段落3
     \______________/          |
          chunk1            chunk2
```

**优势：** 切分边界更符合语义逻辑
**劣势：** 需要额外的 Embedding API 调用

#### Late Chunking

先对整个文档做长上下文 Embedding，再按位置切分向量：

```
文档全文 → Long-context Embed (8K tokens) → 按 chunk 位置切分向量
```

**优势：** 每个切片的向量包含全文上下文信息
**劣势：** 需要支持长上下文的 Embedding 模型

#### Parent-Child Chunking

双层切分策略：

```
Parent Chunk (2048 chars)  ← 检索时返回更大的上下文
├── Child Chunk (512 chars)  ← 用于向量化和匹配
├── Child Chunk (512 chars)
└── Child Chunk (512 chars)
```

**优势：** 匹配精度高（小切片），上下文丰富（返回大切片）
**劣势：** 存储量翻倍，实现复杂度高

### 4.3 LLM 集成层

当前平台定位为纯检索中台，不包含 LLM。未来可添加可选的 LLM 层：

```
Client → RAG Middleware → Retrieve (现有)
                        → LLM Generate (新增, 可选)
                            ├── 基于 source_nodes 生成答案
                            ├── Streaming 输出
                            └── 引用溯源
```

**实现方式：** 新增 `/api/v1/chat` 端点，接受 `model` 参数指定 LLM：
- 内部调用现有 Retrieve 获取 source_nodes
- 将 source_nodes 作为上下文注入 System Prompt
- 通过 LLM API 生成回答
- SSE 流式返回

### 4.4 融合策略增强

当前使用 Qdrant 原生 RRF，未来可探索：

#### DBSF（Distribution-Based Score Fusion）

基于各路检索分数的统计分布进行归一化后融合：

```
normalized_score_i = (score_i - mean_i) / std_i
final_score = Σ weight_i * normalized_score_i
```

**优势：** 考虑分数分布，比 RRF 的纯排名融合更精确。

#### Learned Fusion

训练一个轻量级模型学习最优融合权重：

```
features = [dense_score, sparse_score, bm25_score, query_length, ...]
fusion_score = MLP(features)
```

**优势：** 自适应不同查询类型
**劣势：** 需要标注数据训练

### 4.5 增量索引

**问题：** 当前文档更新需要全量重新向量化。

**方案：** 基于文件 diff 的增量更新：

1. 解析新版本文档
2. 对比新旧切片（基于文本 hash）
3. 只对变化的切片执行 Embed + Upsert
4. 删除已不存在的切片

**收益：** 大文档小修改时，显著减少处理时间和 API 调用。

| # | 债务 | 影响 | 优先级 |

| 7 | 无 API 限流 | 可能被恶意请求打满 | 中 |
| 8 | 无 E2E 测试 | 回归风险 | 低 |
| 9 | 前端无单元测试 | 组件回归风险 | 低 |

## 6. 优先级矩阵

```
                 高价值
                   │
   ┌───────────────┼───────────────┐
   │               │               │
   │  API Key 认证  │  Embedding   │
   │  健康检查      │    缓存       │
   │  Payload 索引  │  查询日志     │
   │               │               │
低努力 ────────────┼──────────────── 高努力
   │               │               │
   │  API 限流      │  PostgreSQL  │
   │  E2E 测试     │  分布式Qdrant │
   │               │  RBAC 权限    │
   │               │  Prometheus  │
   │               │               │
   └───────────────┼───────────────┘
                   │
                 低价值
```

**推荐实施顺序：**

1. Payload 索引 + API Key 认证 + 健康检查（短期，低风险高收益）
2. Embedding 缓存 + 查询日志（短期，中等工作量）
3. PostgreSQL 迁移 + 分布式 Qdrant（中期，为生产扩展铺路）
4. RBAC + Prometheus（中期，运营支撑）
5. 多模态 + 高级切分 + LLM 集成（长期，产品差异化）
