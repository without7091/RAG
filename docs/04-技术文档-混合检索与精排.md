完整检索链路由 `RetrievalService.retrieve()` 编排（`backend/app/services/retrieval_service.py:79-168`）：

```python
async def retrieve(self, knowledge_base_id, query, top_k=20, top_n=3, min_score=None):
    # Step 1: 三路查询向量化
    dense_vector = await self.embedding.embed_query(query)
    sparse_vector = await self.sparse_embedding.embed_query_async(query)
    bm25_vector = self.bm25.text_to_sparse_vector(query) if self.bm25 else None

    # Step 2: 混合检索
    candidates = await self.vector_store.hybrid_search(
        collection_name=knowledge_base_id,
        dense_vector=dense_vector,
        sparse_vector=sparse_vector,
        top_k=top_k,
        bm25_vector=bm25_vector,
    )

    # Step 3: 精排
    texts = [hit["payload"].get("text", "") for hit in candidates]
    reranked = await self.reranker.rerank(query, texts, top_n=top_n)

    # Step 4: 构建结果 + min_score 过滤
    # Step 5: 上下文合成
    source_nodes = await synthesize_context(source_nodes, knowledge_base_id, self.vector_store)
```

## 2. 三路向量架构

### 2.1 Dense 向量 — 语义检索

**模型：** Qwen3-Embedding-4B
**维度：** 1024
**距离度量：** COSINE

Dense 向量将查询和文档映射到连续的高维语义空间。通过余弦相似度计算语义接近程度。

**优势：**
- 理解同义词和近义表达（"机器学习" ≈ "ML"）
- 捕捉段落级别的主题相似性
- 对自然语言查询效果好

**劣势：**
- 对精确关键词（编号、专有名词）不敏感
- 可能将语义相似但实际无关的内容排到前面

### 2.2 Learned Sparse (SPLADE) — 学习型稀疏检索

**模型：** Qdrant/bm25（FastEmbed SPLADE 模型）
**索引修饰器：** IDF

SPLADE 通过神经网络学习词项权重，比传统 BM25 更智能：

**优势：**
- 自动学习词项重要性（比 TF-IDF 更准确）
- 能扩展查询到相关的同义词项
- 稀疏表示，高效索引

**劣势：**
- 需要额外的模型文件（~100MB）
- 对未见过的专业术语泛化能力有限

### 2.3 BM25 — 传统关键词检索

**实现：** 自研 BM25Service（Jieba + CRC32）
**索引修饰器：** IDF（Qdrant 服务端计算）

```python
# 分词流程
text = "RAG系统的向量检索性能"
tokens = jieba.lcut(text)  # ["RAG", "系统", "的", "向量", "检索", "性能"]
# 去停用词
tokens = ["RAG", "系统", "向量", "检索", "性能"]  # 去掉"的"
# CRC32 哈希
sparse = {
    crc32("rag") % 2^20: 1.0,
    crc32("系统") % 2^20: 1.0,
    crc32("向量") % 2^20: 1.0,
    crc32("检索") % 2^20: 1.0,
    crc32("性能") % 2^20: 1.0,
}
```

**优势：**
- 精确匹配（编号、人名、产品型号）
- 无模型依赖，纯算法实现
- 计算速度极快

**劣势：**
- 无语义理解能力
- 受分词质量影响

### 三路互补分析

| 查询类型 | Dense | SPLADE | BM25 | 最佳策略 |
|---------|-------|--------|------|---------|
| 自然语言问句 | ★★★ | ★★ | ★ | Dense 主导 |
| 精确编号/型号 | ★ | ★★ | ★★★ | BM25 主导 |
| 技术概念查询 | ★★★ | ★★★ | ★★ | Dense+SPLADE |
| 中英混合查询 | ★★ | ★★ | ★★★ | 三路均衡 |
| 短查询（1-2 词） | ★ | ★★ | ★★★ | BM25+SPLADE |
| 长查询（段落级） | ★★★ | ★ | ★ | Dense 主导 |

RRF 融合的价值在于：不需要人工判断查询类型，自动让最匹配的通道发挥作用。

## 3. Qdrant RRF 融合机制

### 三路 Prefetch

```python
# backend/app/services/vector_store_service.py:128-177
async def hybrid_search(self, collection_name, dense_vector, sparse_vector,
                        top_k=10, bm25_vector=None):
    prefetch = [
        models.Prefetch(
            query=dense_vector,
            using="dense",
            limit=top_k,
        ),
    ]

    if sparse_vector.get("indices") and sparse_vector.get("values"):
        prefetch.append(
            models.Prefetch(
                query=models.SparseVector(
                    indices=sparse_vector["indices"],
                    values=sparse_vector["values"],
                ),
                using="sparse",
                limit=top_k,
            ),
        )

    if bm25_vector and bm25_vector.get("indices") and bm25_vector.get("values"):
        prefetch.append(
            models.Prefetch(
                query=models.SparseVector(
                    indices=bm25_vector["indices"],
                    values=bm25_vector["values"],
                ),
                using="bm25",
                limit=top_k,
            ),
        )

    results = await self.client.query_points(
        collection_name=collection_name,
        prefetch=prefetch,
        query=models.FusionQuery(fusion=models.Fusion.RRF),
        limit=top_k,
    )
```

### Reciprocal Rank Fusion (RRF) 原理

RRF 是一种无参数的排名融合算法，核心公式：

```
RRF_score(d) = Σ  1 / (k + rank_i(d))
              i∈{dense, sparse, bm25}
```

其中 `k` 是常数（Qdrant 默认 k=60），`rank_i(d)` 是文档 d 在第 i 路检索结果中的排名。

**特点：**
- **无需归一化**：不同检索通道的分数尺度可以不同
- **排名驱动**：只关心相对顺序，不关心绝对分数
- **自然降级**：如果某一路没有返回某文档，该路的贡献为 0
- **Qdrant 原生支持**：通过 `FusionQuery(fusion=Fusion.RRF)` 在服务端完成

**示例：**

| 文档 | Dense 排名 | Sparse 排名 | BM25 排名 | RRF 得分 (k=60) |
|------|-----------|------------|----------|----------------|
| A | 1 | 3 | 2 | 1/61 + 1/63 + 1/62 = 0.0486 |
| B | 5 | 1 | 1 | 1/65 + 1/61 + 1/61 = 0.0482 |
| C | 2 | - | 5 | 1/62 + 0 + 1/65 = 0.0315 |

文档 A 和 B 分数接近，但 A 在三路都有不错的排名，更可能是全面相关的结果。

## 4. Reranker 精排

### 为什么需要 Reranker

粗排（Hybrid Search + RRF）的局限：

1. Dense 和 Sparse 是独立编码 query 和 document，缺乏交叉注意力
2. RRF 只看排名不看内容，可能将"巧合高排名"的不相关结果推上去
3. 粗排 top_k=20 中可能只有 3-5 个真正相关的

### Reranker 工作原理

**模型：** Qwen3-Reranker-4B

```
[CLS] query [SEP] document [SEP]
         │                    │
         ▼ ─── 交叉注意力 ──── ▼
                   │
              相关性分数 ∈ [0, 1]
```

Reranker 将 query 和 document 拼接后送入 Transformer，通过交叉注意力机制让 query 中的每个 token 与 document 中的每个 token 直接交互，产生更精准的相关性判断。

### API 调用

```python
# backend/app/services/reranker_service.py
class RerankerService:
    @retry_on_api_error(max_attempts=3)
    async def rerank(self, query: str, texts: list[str], top_n: int = 3) -> list[dict]:
        client = await self._get_client()
        payload = {
            "model": self.model,           # "Qwen/Qwen3-Reranker-4B"
            "query": query,
            "documents": texts,
            "top_n": top_n,
            "return_documents": False,
        }
        response = await client.post(self.api_url, json=payload, headers=headers)
        # 返回: [{"index": int, "score": float}, ...]
```

### 精排 vs 粗排的效果对比

典型场景：查询 "如何优化数据库查询性能"

| 排名 | 粗排结果 (RRF) | 精排结果 (Reranker) |
|-----|-------------|------------------|
| 1 | "数据库查询日志配置" (关键词命中) | "数据库查询性能优化最佳实践" |
| 2 | "数据库索引创建步骤" (部分相关) | "SQL 查询优化的 10 个技巧" |
| 3 | "查询性能监控工具介绍" (部分相关) | "索引策略与查询计划分析" |

精排通过交叉注意力能理解"优化+性能"的组合语义，而粗排可能被"查询"这个高频词误导。

## 5. min_score 过滤

精排后根据分数阈值过滤低质量结果：

```python
# backend/app/services/retrieval_service.py:151-155
effective_min_score = (
    min_score if min_score is not None
    else get_settings().reranker_min_score  # 默认 0.1
)
if effective_min_score > 0:
    source_nodes = [
        n for n in source_nodes if n["score"] >= effective_min_score
    ]
```

### 参数说明

| 参数 | 来源 | 默认值 | 说明 |
|------|------|-------|------|
| `min_score` | 请求参数 | None | 用户指定的阈值 |
| `reranker_min_score` | Settings | 0.1 | 全局默认阈值 |

### 过滤策略

- **min_score=0.1**（默认）：过滤掉 Reranker 认为完全不相关的结果
- **min_score=0.3**：较严格，只保留中等以上相关度的结果
- **min_score=0.0**：不过滤，返回所有精排结果（调试用）
- **min_score=None**：使用全局默认值

## 6. 上下文窗口合成

### 目标

检索到的切片可能只包含答案的一部分。通过拼接相邻切片，为下游 LLM 提供更完整的上下文。

### 实现

```python
# backend/app/services/retrieval_service.py:13-57
async def synthesize_context(source_nodes, knowledge_base_id, vector_store):
    # 1. 收集所有涉及的 doc_id
    doc_ids = {n["doc_id"] for n in source_nodes if n.get("doc_id")}

    # 2. 为每个 doc_id 从 Qdrant 获取所有切片
    doc_chunk_map: dict[str, dict[int, str]] = {}
    for doc_id in doc_ids:
        chunks = await vector_store.get_chunks_by_doc_id(
            knowledge_base_id, doc_id
        )
        doc_chunk_map[doc_id] = {c["chunk_index"]: c["text"] for c in chunks}

    # 3. 为每个结果拼接 [前一个, 当前, 后一个] 切片
    for node in source_nodes:
        chunk_index = node.get("chunk_index")
        chunk_map = doc_chunk_map.get(node["doc_id"], {})

        parts = []
        for ci in (chunk_index - 1, chunk_index, chunk_index + 1):
            if ci in chunk_map:
                parts.append(chunk_map[ci])

        node["context_text"] = "\n\n".join(parts) if parts else node["text"]
```

### 示例

假设文档被切分为 5 个切片（chunk_index 0-4），检索到 chunk_index=2：

```
chunk_0: "第一章 概述..."
chunk_1: "产品功能A: ..."      ← 前一个切片
chunk_2: "产品功能B: ..."      ← 命中切片 (score=0.92)
chunk_3: "产品功能C: ..."      ← 后一个切片
chunk_4: "第二章 安装..."
```

合成后的 `context_text`：
```
产品功能A: ...

产品功能B: ...

产品功能C: ...
```

### 设计考量

- **窗口大小固定为 ±1**：平衡上下文丰富度和 token 消耗
- **按 doc_id 批量获取**：减少 Qdrant 查询次数（相同文档的多个命中共享一次查询）
- **text 字段保留原值**：`context_text` 是扩展上下文，原始 `text` 不变
- **边界安全**：chunk_index=0 时没有前一个，chunk_index=最后时没有后一个，自动跳过

## 7. 参数调优建议

### top_k（粗排候选数）

| 场景 | 建议值 | 说明 |
|------|-------|------|
| 快速原型 | 10 | 响应快，但可能遗漏相关结果 |
| 标准场景 | 20 (默认) | 平衡召回率和速度 |
| 高召回需求 | 50 | 适合文档量大、查询模糊的场景 |
| 精确查询 | 10 | 查询明确时不需要太多候选 |

### top_n（精排返回数）

| 场景 | 建议值 | 说明 |
|------|-------|------|
| 单答案场景 | 1-2 | 只需最佳匹配 |
| 标准 QA | 3 (默认) | 提供 Top-3 供 LLM 参考 |
| 综合分析 | 5-10 | 需要多角度信息 |

### min_score（最低分数阈值）

| 场景 | 建议值 | 说明 |
|------|-------|------|
| 宽松匹配 | 0.05 | 适合探索性查询 |
| 标准场景 | 0.1 (默认) | 过滤掉明显不相关的 |
| 严格匹配 | 0.3 | 只要高置信度结果 |
| 调试模式 | 0.0 | 查看所有精排结果的分数分布 |

### chunk_size（切片大小）

| 场景 | 建议值 | 说明 |
|------|-------|------|
| 高精度检索 | 512 | 切片小，匹配更精确 |
| 标准场景 | 1024 (默认) | 平衡精度和上下文 |
| 长文档概述 | 2048 | 切片大，保留更多上下文 |
| 代码文档 | 1536 | 代码函数通常较长 |

### chunk_overlap（切片重叠）

| 场景 | 建议值 | 说明 |
|------|-------|------|
| 默认 | 128 | chunk_size 的 ~12.5% |
| 高连贯性需求 | chunk_size * 0.2 | 20% 重叠 |
| 节省存储 | 0 | 无重叠（可能丢失边界信息） |

### 组合建议

**高精度场景（法律/合规文档）：**
```json
{"top_k": 30, "top_n": 5, "min_score": 0.2, "chunk_size": 768}
```

**快速问答场景：**
```json
{"top_k": 10, "top_n": 3, "min_score": 0.1, "chunk_size": 1024}
```

**技术文档场景：**
```json
{"top_k": 20, "top_n": 3, "min_score": 0.1, "chunk_size": 1536}
```
