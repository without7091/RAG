# v2 技术报告（四）：v3 风险清单、潜在问题与改进建议

## 1. 当前审阅结论

本轮结论来自三部分事实：

- 代码静态审阅：`backend/app/`、`frontend/src/`、`docs/`
- 接口与契约审阅：FastAPI OpenAPI、Schema、前端 API 类型
- 现状验证：
  - `cd backend && uv run pytest -q`：**因 `tests/unit/test_manual_retrieve_test.py` 引用缺失文件而在收集阶段失败**
  - `cd backend && uv run pytest . --ignore=tests/unit/test_manual_retrieve_test.py -q`：**260 passed, 1 failed**
  - `cd frontend && npm.cmd run lint`：**通过**
  - `cd frontend && npm.cmd run build`：**通过**

因此，当前 v2 的主要风险不是“主链路不可用”，而是：

- 安全与接口治理尚未补齐；
- 少量一致性风险已潜伏在代码路径中；
- 自动化测试存在回归信号。

---

## 2. 高优先级风险

### 风险 1：缺少真正的认证、授权与租户隔离

- **现象/证据**
  - `backend/app/api/v1/` 下所有核心接口均未接入认证依赖
  - `backend/app/schemas/retrieve.py` 中 `user_id` 只是请求字段
  - `backend/app/main.py` 中 CORS 允许全部来源
- **影响**
  - 任何拿到 `knowledge_base_id` 的调用方都可直接上传、删除、检索、下载
  - “多租户”更多是数据组织概念，而不是安全隔离概念
- **触发条件**
  - API 对外暴露到团队外、网关外或公网
- **优先级**
  - 高
- **v3 建议**
  - 引入鉴权中间件或网关鉴权
  - 在知识库/目录/文档层增加所有者或租户字段
  - 将 `user_id` 升级为真实身份上下文
  - 为删除、下载、重试等操作补审计日志

### 风险 2：检索接口的 OpenAPI 契约与真实行为不一致

- **现象/证据**
  - `POST /api/v1/retrieve` 在代码中可返回 `EventSourceResponse`
  - `app.openapi()` 仍把 `200` 响应描述为 `application/json + RetrieveResponse`
  - `stream` 默认值是 `true`
- **影响**
  - 外部系统按 OpenAPI 自动生成客户端时，极易把流式接口误当普通 JSON 接口
  - SSE 行为、`error` 事件、状态事件不会出现在自动文档中
- **触发条件**
  - 第三方系统依赖 `/openapi.json` 或 Swagger 自动接入
- **优先级**
  - 高
- **v3 建议**
  - 将 JSON 模式和 SSE 模式拆成两个显式接口，或至少拆分响应描述
  - 在 OpenAPI 中补齐 `text/event-stream`
  - 显式补充 `400/404/422/500/502` 错误响应模型

### 风险 3：同一知识库下同名文件会覆盖物理源文件

- **现象/证据**
  - `backend/app/api/v1/document.py` 将上传文件写入 `upload_dir/<kb_id>/<file_name>`
  - 物理存储键使用的是原始文件名，而不是 `doc_id`
- **影响**
  - 同一知识库中上传“不同内容但同名”的文件时，旧文件会被覆盖
  - 老文档的“下载”与“重试”可能读到新文件内容
- **触发条件**
  - 用户重复上传同名文件，如 `manual.pdf`
- **优先级**
  - 高
- **v3 建议**
  - 文件物理名改用 `doc_id` 或 UUID
  - 原始文件名仅保存在 metadata 中用于展示
  - 下载时通过逻辑名映射回原始文件名

### 风险 4：重复上传同内容文档可能遗留旧向量

- **现象/证据**
  - `DocumentService.create()` 对同 `(knowledge_base_id, doc_id)` 的文档执行重置
  - 重置后状态会变成 `uploaded`
  - `vectorize_documents()` 只有在文档当前状态是 `completed` 时才设置 `needs_vector_cleanup`
- **影响**
  - 如果同内容文档已完成过向量化，重新上传后再向量化，旧向量可能未被清理
  - Qdrant 中可能出现重复 chunk 点位
- **触发条件**
  - 同一知识库重复上传相同内容文件
- **优先级**
  - 高
- **v3 建议**
  - 在 `DocumentService.create()` 检测到“重置已有文档”时主动设置 `needs_vector_cleanup`
  - 或者在上传时先删旧向量，再重置文档
  - 为重复上传场景补回归测试

## 3. 中优先级风险

### 风险 5：批量向量化是逐文档提交，存在部分成功、部分失败

- **现象/证据**
  - `POST /api/v1/document/vectorize` 在循环中逐条 `commit`
- **影响**
  - 批量请求中如果中途某个文档出错，前面文档已被改为 `pending`
  - 调用方会遇到“请求失败但部分文档已启动处理”的非原子语义
- **触发条件**
  - 一次提交多个 `doc_ids`，其中部分文档不存在或源文件缺失
- **优先级**
  - 中
- **v3 建议**
  - 增加 `all_or_nothing` 事务模式，或明确返回逐项结果
  - 统一定义批量接口的部分成功契约

### 风险 6：`get_chunks_by_doc_id(limit=200)` 会截断大文档视图

- **现象/证据**
  - `backend/app/services/vector_store_service.py` 中 `limit` 默认是 `200`
- **影响**
  - 大文档 chunk 预览可能不完整
  - 上下文合成如果依赖完整 chunk 列表，超大文档上存在不完整窗口风险
- **触发条件**
  - 文档 chunk 数量超过 `200`
- **优先级**
  - 中
- **v3 建议**
  - 改为循环 scroll 直到取全
  - 前台预览接口改分页返回

### 风险 7：知识库列表存在 N+1 查询

- **现象/证据**
  - `list_kbs()` 对每个知识库单独调用 `get_document_count()`
- **影响**
  - 知识库数量一多，列表接口性能会线性变差
- **触发条件**
  - 知识库规模上升到几十或上百
- **优先级**
  - 中
- **v3 建议**
  - 改为单次聚合查询返回 `document_count`
  - 树接口与列表接口共用统计缓存或聚合 SQL

### 风险 8：Worker 重试状态仅保存在进程内存

- **现象/证据**
  - `PipelineWorker` 的 `_retry_counts` 是内存字典
- **影响**
  - 服务重启后重试次数丢失
  - 多实例部署时无法共享任务执行状态
- **触发条件**
  - 需要横向扩容、重启恢复或跨实例调度
- **优先级**
  - 中
- **v3 建议**
  - 在 `documents` 表或独立任务表中持久化重试计数
  - 增加 lease/owner/heartbeat 字段，避免多实例重复消费

### 风险 9：SSE 实现缺少心跳、事件版本与重连语义

- **现象/证据**
  - 当前仅发送 `status/result/error`
  - 前端使用手写 parser 解析 SSE 行
- **影响**
  - 经过网关或代理时，长连接更容易被中间层切断
  - 客户端难以实现断线重连与幂等恢复
- **触发条件**
  - 复杂网络环境、反向代理、超时严格的网关
- **优先级**
  - 中
- **v3 建议**
  - 增加心跳事件
  - 引入 `request_id/event_id`
  - 为流式模式定义客户端重连建议

### 风险 10：读取型接口带有迁移副作用

- **现象/证据**
  - `KBService.list_all()` 与 `list_tree()` 会触发 `assign_default_folder_to_unassigned()`
- **影响**
  - 单纯的读取请求会修改数据库状态
  - 不利于严格审计与只读副本场景
- **触发条件**
  - 系统升级后首次调用列表/树接口
- **优先级**
  - 中
- **v3 建议**
  - 把默认目录回填迁移到显式迁移脚本或启动脚本
  - 让读取接口保持只读语义

## 4. 低到中优先级风险

### 风险 11：前后端状态类型存在漂移

- **现象/证据**
  - 前端 `DocumentStatus` 额外包含 `initializing`
  - 后端 schema 不返回该状态
- **影响**
  - 如果外部系统直接参考前端类型，容易误解后端契约
- **触发条件**
  - 以 `frontend/src/lib/api.ts` 作为唯一接口依据
- **优先级**
  - 低到中
- **v3 建议**
  - 在 API 文档中明确标注 `initializing` 为前端临时态
  - 把 UI 临时态从共享类型中剥离

### 风险 12：自动化测试存在明显回归信号

- **现象/证据**
  - `tests/unit/test_manual_retrieve_test.py` 依赖缺失的 `manual_retrieve_test.py`
  - 忽略该文件后，`tests/unit/test_pipeline_worker.py::test_does_not_retry_non_retryable_error` 仍失败
- **影响**
  - CI 稳定性下降
  - 难以快速判断后续改动是否引入真实回归
- **触发条件**
  - 跑完整后端测试集
- **优先级**
  - 低到中
- **v3 建议**
  - 清理失效测试依赖
  - 修复 `PipelineWorker` 非可重试错误分支的测试或实现
  - 将“主链路必过测试”与“工具脚本测试”拆开

### 风险 13：OpenAPI 对错误码覆盖不足

- **现象/证据**
  - 路由层多数仅声明 `200/422`
  - 实际异常处理支持 `400/404/409/500/502`
- **影响**
  - 第三方系统无法从自动文档准确获知错误面
- **触发条件**
  - 自动生成 SDK、错误码联调
- **优先级**
  - 低到中
- **v3 建议**
  - 为核心接口补全 `responses=...`
  - 提供统一错误响应模型

## 5. 建议的 v3 改进优先级

### P0：上线前必须补齐

1. 认证授权与租户隔离
2. 检索接口 JSON/SSE 契约拆分
3. 文件存储命名与重复上传向量一致性

### P1：平台化必须补齐

1. Worker 状态持久化
2. 批量接口原子性或逐项结果契约
3. SSE 心跳与 request_id
4. 自动化测试回归修复

### P2：规模化时优先处理

1. 知识库列表聚合优化
2. chunk 预览分页/scroll 全量拉取
3. 读接口去副作用化

---

## 6. 总体建议

当前 v2 的主链路已经足够支撑产品验证和内部接入，但如果目标是 v3 面向更多系统复用，优先级应该从“继续加功能”切换到“先补平台底座”：

- 先补 **安全**
- 再补 **契约**
- 再补 **一致性**
- 最后补 **规模化**

也就是说，v3 最值得投入的方向不是再堆更多检索技巧，而是让当前已经不错的 v2 能力，变成真正稳定、清晰、可治理的对外平台。
